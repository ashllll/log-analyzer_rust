# 搜索与过滤

搜索页将查询表达式与结构化过滤器一起提交。后端先验证查询，再建立执行计划，并把结果写入可分页的搜索会话。

## 查询写法

### 单关键词

```text
connection refused
```

### OR 查询

使用 `|` 分隔多个候选词：

```text
timeout|retry|circuit breaker
```

### 正则与 look-around

标准表达式由 Rust `regex` 引擎处理；需要 look-around 的模式会进入 Fancy 引擎。查询验证器会在执行前拒绝高风险的嵌套量词或重叠交替模式，以降低 ReDoS 风险。

## 过滤器

| 过滤器 | 用途 | 示例 |
| --- | --- | --- |
| Level | 限定日志级别 | 只看 `E` 和 `W` |
| Time Range | 限定事件时间 | 事故窗口前后 15 分钟 |
| File Pattern | 限定文件名 / 虚拟路径 | `gateway/*.log`、`error*.txt` |

## 高信号搜索顺序

1. 先用 2–4 个与故障强相关的关键词。
2. 限定事故时间窗口。
3. 再按日志级别收敛。
4. 已知组件或文件位置时补充 File Pattern。
5. 打开单条结果查看完整上下文与虚拟路径。

## 分页、取消与生命周期

每次搜索返回一个 `search_id`。前端使用它拉取分页结果，也可以取消仍在执行的会话。结果存储在磁盘而不是一次性返回到 UI，适合命中数量很大的查询。

::: tip 查询太慢？
优先缩短时间范围或限制文件模式。过宽的正则会增加每行匹配成本，即使它通过安全校验。
:::

更深入的执行细节见[搜索架构](../architecture/search.md)。
