> 日期：2026-04-23

> 关联文档：`PURE_RUST_STORAGE_MIGRATION_PLAN.md`、`ARCHIVE_SEARCH_PIPELINE_ANALYSIS.md`

## 1. 当前状况审计

### 1.1 SQLite 的跨平台现状

通过 `cargo tree` 确认，当前 `libsqlite3-sys` 已启用 **`bundled`** feature：

```
libsqlite3-sys v0.30.1 bundled,bundled_bindings,cc,pkg-config,unlock_notify,vcpkg
```

这意味着：
- ✅ **Windows**：`bundled` 模式会自动下载 SQLite C 源码并用 `cc` crate 编译，无需系统预装 SQLite 或 vcpkg
- ✅ **macOS**：同样自动编译，无需 Homebrew 安装 SQLite
- ✅ **Linux**：同样自动编译，无需 `libsqlite3-dev`

**结论：当前 SQLite 的跨平台编译问题实际上已经通过 `bundled` 解决了。**

### 1.2 真正的痛点是什么？

如果跨平台编译已经不是问题，那么"去掉 SQLite C 依赖"的诉求可能来自：

| 痛点 | 严重程度 | 说明 |
|------|:--------:|------|
| 编译时间 | 中 | `cc` 编译 SQLite C 代码增加约 10-20s 编译时间 |
| 安全审计面 | 中 | C 代码引入内存安全风险（尽管 SQLite 质量极高） |
| 依赖链复杂 | 低 | `sqlx` → `sqlx-sqlite` → `libsqlite3-sys` → `cc`/`pkg-config`/`vcpkg` |
| 哲学偏好 | 低 | "全栈 Rust" 的工程美学 |

## 2. 候选方案跨平台评估

### 2.1 评估维度

| 维度 | 权重 | 说明 |
|------|:----:|------|
| 跨平台编译 | 🔴 高 | Windows 是否"开箱即用" |
| 运行时性能 | 🔴 高 | 导入、搜索、查询速度 |
| 兼容性/成熟度 | 🔴 高 | 生产环境验证、格式稳定性、社区规模 |
| 纯 Rust | 🟡 中 | 用户明确偏好，但不是唯一约束 |
| 编译速度 | 🟡 中 | 开发体验 |
| 二进制体积 | 🟢 低 | 桌面应用不敏感 |

### 2.2 候选方案对比

#### A. 保留 SQLite（当前方案，优化使用方式）

| 维度 | 评分 | 说明 |
|------|:----:|------|
| 跨平台编译 | ⭐⭐⭐⭐⭐ | `bundled` 已解决所有平台编译问题 |
| 运行时性能 | ⭐⭐⭐⭐⭐ | WAL 模式，桌面场景足够 |
| 兼容性 | ⭐⭐⭐⭐⭐ | 25 年历史，数十亿设备验证 |
| 纯 Rust | ⭐ | 底层是 C |
| 编译速度 | ⭐⭐⭐ | C 编译增加 ~15s |

**优点**：零迁移风险、零学习成本、生态系统最丰富、FTS5 全文索引、外键约束、事务、多索引原生支持。

**缺点**：不是纯 Rust、C 依赖增加安全审计面。

**适用场景**：如果"兼容性"和"零风险"是最高优先级。

---

#### B. redb（纯 Rust B-tree）

| 维度 | 评分 | 说明 |
|------|:----:|------|
| 跨平台编译 | ⭐⭐⭐⭐⭐ | 纯 Rust，cargo 一键通过 |
| 运行时性能 | ⭐⭐⭐⭐ | B-tree 读性能优异（零拷贝），写与 SQLite 相当 |
| 兼容性 | ⭐⭐⭐ | 较新（2022 年发布），桌面应用大规模验证不足；1.x→2.x 有格式 breaking change |
| 纯 Rust | ⭐⭐⭐⭐⭐ | 100% Rust |
| 编译速度 | ⭐⭐⭐⭐⭐ | 无 C 编译开销 |

**优点**：纯 Rust、编译最简单、零拷贝读取、MVCC 并发读不阻塞写、单文件。

**缺点**：
- 成熟度不如 SQLite（但比 sled 好很多）
- 无内置全文索引（FTS5 替代需自行实现）
- 无外键约束（需应用层实现）
- 写事务同步（需在 `spawn_blocking` 中执行）
- **格式稳定性风险**：历史版本间有 breaking change，未来升级可能需要数据迁移

**适用场景**：如果"纯 Rust"和"跨平台编译极简"优先级高于"十年级成熟度"。

---

#### C. heed / LMDB（C 封装，但极稳定）

| 维度 | 评分 | 说明 |
|------|:----:|------|
| 跨平台编译 | ⭐⭐⭐⭐ | LMDB 是 C 库，但非常稳定，所有平台支持良好 |
| 运行时性能 | ⭐⭐⭐⭐⭐ | 内存映射，读性能极佳（比 SQLite 快 2-5x） |
| 兼容性 | ⭐⭐⭐⭐⭐ | LMDB 10+ 年历史（OpenLDAP 项目），heed 由 Meilisearch 维护 |
| 纯 Rust | ⭐⭐ | 底层是 C |
| 编译速度 | ⭐⭐⭐⭐ | C 代码量小，编译快于 SQLite |

**优点**：
- 读性能极佳（mmap）
- 单文件、零配置
- 跨平台极其稳定（iOS/Android/BSD/Windows/macOS/Linux）
- 写入是追加式，不会损坏已有数据
- 崩溃安全（copy-on-write）

**缺点**：
- 不是纯 Rust
- 写事务单线程（与 redb 相同限制）
- 数据库文件大小只能增长（需要手动 compact）
- 无全文索引

**适用场景**：如果"读性能"和"稳定性"是最高优先级，可以接受非纯 Rust。

---

#### D. native_db（基于 redb 的 ORM）

| 维度 | 评分 | 说明 |
|------|:----:|------|
| 跨平台编译 | ⭐⭐⭐⭐⭐ | 纯 Rust |
| 运行时性能 | ⭐⭐⭐⭐ | 基于 redb，性能接近 |
| 兼容性 | ⭐⭐⭐ | 基于 redb，额外增加一层抽象风险；API 标注为 unstable |
| 纯 Rust | ⭐⭐⭐⭐⭐ | 100% Rust |

**优点**：ORM 风格、自动索引管理、有 Tauri 示例。

**缺点**：API 不稳定、macro 增加编译时间、对复杂查询支持有限。

**适用场景**：如果团队偏好 ORM 风格且项目规模不大。

---

#### E. sled（已归档，排除）

| 维度 | 评分 | 说明 |
|------|:----:|------|
| 兼容性 | ⭐ | **2023 年停止维护**，不推荐使用 |

## 3. 关键决策点

### 3.1 问题："兼容性不能太差"意味着什么？

这是决策的核心。请确认以下优先级：

**场景 1：兼容性 > 纯 Rust**（推荐：保留 SQLite 或 heed/LMDB）
- 你的应用面向企业用户，数据不能丢
- 需要 5 年以上长期维护
- 不能有"数据库格式升级导致数据丢失"的风险

**场景 2：纯 Rust ≈ 兼容性**（推荐：redb + SQLite fallback）
- 你愿意接受 redb 的成熟度，但希望有退路
- 可以通过 feature flag 保留 SQLite

**场景 3：纯 Rust > 兼容性**（推荐：redb）
- 你的应用是消费级工具，数据可重建
- 你愿意承担 redb 格式升级时的迁移工作
- "全栈 Rust"是核心工程目标

### 3.2 一个被忽视的事实

当前 SQLite（bundled）在跨平台上**已经工作良好**。如果用户的真实痛点不是"跨平台编译失败"，那么替换 SQLite 的收益可能不如预期：

| 收益 | 实际价值 |
|------|----------|
| 去掉 C 依赖 | 中（SQLite C 代码质量极高，安全风险极低） |
| 编译更快 | 低（C 编译增加 ~15s，在 Tauri 整体编译中占比 <5%） |
| 二进制更小 | 低（桌面应用不敏感） |
| 安全审计更简单 | 中（但 SQLite 是极少数通过形式化验证的 C 项目） |

## 4. 最终推荐方案

### 4.1 推荐：双后端架构（redb 为主，SQLite 为 fallback）

如果必须在"纯 Rust"和"兼容性"之间取得平衡，最佳方案是：**不要二选一，而是两者都要。**

```
                ┌─────────────────────┐
                │   MetadataStorage   │
                │       trait         │
                └──────────┬──────────┘
                           │
           ┌───────────────┼───────────────┐
           ↓               ↓               ↓
    ┌─────────────┐ ┌─────────────┐ ┌────────────┐
    │  RedbMetadata│ │SqliteMetadata│ │  MockStore │
    │    Store     │ │    Store     │ │  (tests)   │
    │   (默认)      │ │  (fallback)  │ │            │
    └─────────────┘ └─────────────┘ └────────────┘
```

**实施策略**：

1. **Phase 1**：扩展 `MetadataStorage` trait，让业务代码与具体后端解耦
2. **Phase 2**：实现 `RedbMetadataStore`，在开发/测试环境验证
3. **Phase 3**：发布时默认使用 redb，但保留编译开关 `-F sqlite-backend`
4. **Phase 4**：收集 3-6 个月用户反馈，如果 redb 稳定则彻底切换；如果有问题则回退到 SQLite

**为什么这是最佳平衡**：
- 满足了"纯 Rust"的诉求
- 通过 trait 抽象保留了回退能力
- 不押注单一后端，风险可控
- 如果 redb 表现良好，未来可以去掉 SQLite 代码；如果不行，用户无感知

### 4.2 替代推荐：如果不替换 SQLite，最低限度优化

如果评估后认为替换 SQLite 的 ROI 不够高，那么至少应该做以下优化：

1. **去掉无效的 Tantivy 索引重建**（最大浪费）
2. **清理解压后的临时目录**（磁盘占用翻倍）
3. **简化 SQLite schema**（去掉不用的 FTS5、index_state 表）
4. **考虑用 `rusqlite` 替代 `sqlx`**（更轻量，编译更快，API 更简单）

`rusqlite` 相比 `sqlx` 的优势：
- 同步 API，不需要 `async_trait` 和 `tokio` 运行时适配
- `bundled` feature 更成熟
- 编译产物更小
- 对于桌面应用的同步/简单查询场景，`sqlx` 的 async 复杂度是过度设计

## 5. 决策检查清单

在做出最终决定前，请确认以下问题：

| # | 问题 | 如果选择"是" | 如果选择"否" |
|---|------|-------------|-------------|
| 1 | 当前 SQLite 在 Windows 上编译有问题吗？ | 先解决编译问题（通常是 `bundled` feature） | 替换动机 -1 |
| 2 | 是否有强制的安全审计要求（如政府/金融行业）？ | 纯 Rust 有优势 | 替换动机 -1 |
| 3 | 数据丢失/损坏是否可以接受（可重建）？ | 可以用 redb | 建议保留 SQLite 或 heed |
| 4 | 团队愿意承担 redb 格式升级时的迁移工作吗？ | 可以用 redb | 建议保留 SQLite |
| 5 | 搜索是否需要复杂的全文检索（不只是前缀匹配）？ | SQLite FTS5 有优势，或需引入 Tantivy | redb/heed 需要额外方案 |
| 6 | 导入性能是否是关键瓶颈？ | 考虑 heed（写性能优于 SQLite） | 替换动机 -1 |

## 6. 结论

### 如果必须给出一个单一推荐

> **对于 Tauri 桌面应用（Mac/Windows/Linux），在"通用+性能+兼容性"三角约束下，当前没有纯 Rust 方案能在"兼容性"维度上与 SQLite 或 LMDB 媲美。**

**最务实的路径**：

```
Step 1: 优化当前架构（去掉无效 Tantivy 重建、清理临时文件、修复 bug）
        ↓ ROI 最高，立即可做
Step 2: 扩展 MetadataStorage trait，解耦业务代码与 SQLite
        ↓ 为后端切换做准备
Step 3: 实现 RedbMetadataStore，内部测试验证
        ↓ 风险可控
Step 4: 发布时默认 redb，保留 SQLite feature flag
        ↓ 用户无感知，团队可回退
Step 5: 3-6 个月后评估，决定是否彻底切换
```

**如果等不了 3-6 个月，现在就要决策**：

- **选兼容性**：保留 SQLite（已验证跨平台工作良好）
- **选纯 Rust + 可接受风险**：redb
- **选读性能 + 稳定性**：heed/LMDB
