## 1. 项目概述

### 1.1 项目简介

Log Analyzer 是一款基于 **Rust + Tauri 2.0 + React 19** 技术栈打造的高性能跨平台桌面日志分析工具。专为开发者与运维人员设计，提供快速、可靠的日志搜索与分析能力。

### 1.2 核心特性

| 特性类别 | 具体功能 |
|---------|----------|
| **高性能搜索** | Aho-Corasick 多模式匹配 + Tantivy 全文搜索引擎，响应 <200ms |
| **智能存储** | CAS 内容寻址存储 + SQLite FTS5，自动去重节省 30%+ 空间 |
| **全格式支持** | ZIP/TAR/GZ/RAR/7Z，支持 7 层嵌套压缩包递归解析 |
| **完全离线** | 所有数据本地处理，零网络传输 |
| **插件系统** | 基于 libloading 动态库加载，ABI 版本校验 |

### 1.3 技术栈

| 层级 | 技术选型 |
|------|----------|
| **前端框架** | React 19.1.0 + TypeScript 5.8.3 |
| **状态管理** | Zustand 5.0.9 + @tanstack/react-query 5.90.12 |
| **样式方案** | Tailwind CSS 3.4.17 |
| **后端框架** | Rust 1.70+ + Tauri 2.0 |
| **异步运行时** | tokio 1.x |
| **数据库** | SQLite (sqlx 0.7) |
| **搜索引擎** | Tantivy 0.22 |
| **多模式匹配** | Aho-Corasick 算法 |
| **存储架构** | CAS + SQLite FTS5 |

### 1.4 性能指标

| 指标 | 数值 |
|------|------|
| 搜索吞吐量 | 10,000+ 次/秒 |
| 单关键词搜索 | <10ms |
| 多关键词搜索（10个） | <50ms |
| Tantivy 全文搜索 | <200ms |
| ZIP 解压（100MB） | <5秒 |
| 索引构建（10,000行） | <1秒 |

| 层级 | 职责 | 关键技术 |
|------|------|----------|

### 2.3 数据流

```
用户输入 → SearchPage → SearchQueryBuilder
       ↓
    Tauri invoke('search_logs')
       ↓
    commands/search.rs
       ↓

       ↓
    SearchEngineManager (Tantivy)
       ↓
    Event emit('search-results')
       ↓
    useSearchListeners → 状态更新 → UI 渲染
```

## 3. 核心模块说明

### 3.1 Rust 后端模块

#### 3.1.1 commands/ - 命令层

命令层是 Tauri IPC 接口的实现层，提供前后端通信桥梁。

| 模块 | 功能描述 |
|------|----------|
| `search.rs` | 日志搜索及缓存，支持关键词和正则表达式 |
| `import.rs` | 文件/文件夹导入，支持压缩包递归解析 |
| `workspace.rs` | 工作区管理（创建、加载、删除） |
| `query.rs` | 结构化查询执行 |
| `export.rs` | 搜索结果导出（CSV/JSON） |
| `watch.rs` | 文件监听，增量索引更新 |
| `config.rs` | 配置管理（加载/保存） |
| `performance.rs` | 性能指标获取 |

**命令注册模式**：
```rust
#[tauri::command]
async fn search_logs(
    state: State<'_, AppState>,
    query: String,
    workspace_id: Option<String>,
    // ...
) -> Result<SearchResults, CommandError>
```

#### 3.1.2 search_engine/ - 搜索引擎

基于 Tantivy 实现的高性能全文搜索引擎。

**核心组件**：

| 组件 | 职责 |
|------|------|
| `manager.rs` | 搜索引擎管理器，索引生命周期管理 |
| `schema.rs` | Tantivy schema 定义（content, timestamp, level 等字段） |
| `boolean_query_processor.rs` | 布尔查询处理器（AND/OR/NOT） |
| `highlighting_engine.rs` | 搜索结果高亮引擎 |
| `advanced_features.rs` | 自动补全、过滤器、分区索引 |

**性能优化**：
- 多关键词交集/并集算法加速
- 查询超时 `tokio::time::timeout` 双重保障
- 索引写入独立作用域提交，减少锁持有时间

#### 3.1.3 storage/ - 存储系统

基于 Git 对象存储模型的 CAS（内容寻址存储）系统。

**核心组件**：

| 组件 | 职责 |
|------|------|
| `cas.rs` | CAS 存储核心，SHA-256 哈希寻址 |
| `metadata_store.rs` | SQLite 元数据存储，FTS5 全文索引 |
| `coordinator.rs` | 存储协调器 |
| `integrity.rs` | 数据完整性校验 |

**CAS 特性**：
- SHA-256 哈希作为内容唯一标识
- 扁平目录结构（哈希前两位作为子目录）
- 内存 LRU 缓存（moka）
- 流式存储零拷贝

#### 3.1.4 archive/ - 压缩包处理

支持多种压缩格式的统一处理框架。

**支持格式**：
- ZIP（zip crate）
- TAR/TAR.GZ（tar crate）
- GZIP（flate2 crate）
- RAR（unrar crate）
- 7Z（sevenz-rust crate）

**安全特性**：
- 文件大小限制（默认 100MB）
- 解压后总大小限制（默认 1GB）
- 文件数量限制（默认 1000）
- 路径安全检查（O_NOFOLLOW）
- 嵌套压缩包深度限制

#### 3.1.5 services/ - 业务服务

核心业务逻辑实现。

| 服务 | 职责 |
|------|------|
| `PatternMatcher` | Aho-Corasick 多模式匹配，O(n+m) 复杂度 |
| `QueryExecutor` | 查询执行协调器 |
| `QueryValidator` | 查询验证 |
| `QueryPlanner` | 查询计划构建 |
| `FileWatcher` | 文件系统监听 |

#### 3.1.6 task_manager/ - 任务管理

基于 Actor 模型的异步任务管理。

**核心设计**：
- `TaskManager` 客户端句柄
- `TaskManagerActor` 服务端处理消息
- `ActorMessage` 消息类型枚举

**事件驱动**：通过 `app.emit("task-update")` 向前端推送实时进度。

#### 3.1.7 models/ - 数据模型

定义整个应用的核心数据结构。

| 模型 | 说明 |
|------|------|
| `SearchQuery` | 搜索查询结构 |
| `LogEntry` | 日志条目 |
| `WorkspaceState` | 工作区状态 |
| `AppConfig` | 应用配置 |
| `SearchFilters` | 搜索过滤器 |

## 7. 常见问题

### 7.1 编译问题

#### Q: cargo build 失败，提示缺少依赖

**A**: 确保 Rust 工具链已更新：

```bash
rustup update
cargo fetch
```

#### Q: 前端依赖安装失败

**A**: 清除 npm 缓存并重试：

```bash
npm cache clean --force
rm -rf node_modules package-lock.json
npm install
```

### 7.2 运行时问题

#### Q: 搜索无结果

**A**: 检查以下几点：
1. 工作区是否正确加载
2. 索引是否已构建（查看日志）
3. 查询语法是否正确

#### Q: 导入文件卡住

**A**: 检查文件大小和数量限制，查看任务进度：
```bash
invoke('get_task_info', { taskId: 'xxx' })
```

### 7.3 性能问题

#### Q: 大文件搜索慢

**A**: 启用流式搜索：
- 超过 10,000 条结果自动切换流式模式
- 使用关键词过滤减少搜索范围

#### Q: 内存占用高

**A**: 调整配置减少缓存：
- 减小 LRU 缓存容量
- 限制单次搜索结果数量

### 7.4 安全问题

#### Q: 如何防止路径遍历攻击？

**A**: 代码已实现多层防护：
1. 使用 `O_NOFOLLOW` 标志拒绝符号链接
2. 路径规范化后检查前缀
3. 限制解压文件数量和总大小

#### Q: 数据存储安全吗？

**A**: 所有数据本地存储：
- 零网络传输
- SHA-256 哈希校验完整性
- 无外部依赖
