## 三、性能瓶颈分析

### 3.1 已识别的主要瓶颈

#### 瓶颈1: document_to_log_entry 字段提取
- **位置**: manager.rs:429-527
- **问题**: 17个match链，每字段3-5次提取尝试
- **影响**: 大批量文档转换时CPU开销显著
- **优化建议**: 使用结构化字段提取器

#### 瓶颈2: SnippetGenerator 重复创建
- **位置**: highlighting_engine.rs:252
- **问题**: 每个文档都重新创建SnippetGenerator
- **影响**: 批量高亮时重复初始化开销
- **优化建议**: 复用SnippetGenerator实例

#### 瓶颈3: 时间分区查询
- **位置**: advanced_features.rs:401-416
- **问题**: 范围查询需遍历所有重叠分区
- **影响**: 跨年查询时可能扫描数百个分区
- **优化建议**: 预计算分区覆盖Bitmap

#### 瓶颈4: Trie深度遍历
- **位置**: advanced_features.rs:527-544
- **问题**: BFS遍历受限于max_suggestions*2
- **影响**: 深Trie前缀查询延迟增加
- **优化建议**: 限制遍历深度

### 3.2 锁竞争分析

| 锁类型 | 位置 | 持有时间 | 并发影响 |
|-------|------|---------|---------|
| parking_lot::RwLock | manager.rs reader | <1ms | 高并发无竞争 |
| parking_lot::RwLock | term_stats | <100μs | 中等并发影响 |
| tokio::sync::Mutex | performance_monitor | <1ms | 高并发无竞争 |
| parking_lot::Mutex | streaming progress | <1μs | 无竞争 |

## 五、优化建议

### 5.1 短期优化（<1天）

#### 优化1: SnippetGenerator实例复用
```rust
// 当前: 每个文档创建新实例
let mut snippet_generator = SnippetGenerator::create(&searcher, query, self.content_field)?;

// 优化: 缓存SnippetGenerator实例
struct HighlightingEngine {
    snippet_generator_cache: Arc<RwLock<LruCache<u64, SnippetGenerator>>>,
    // ...
}
```

#### 优化2: 批量字段提取器
```rust
// 当前: 17个match链
let content = match doc.get_first(schema.content) { ... };

// 优化: 结构化提取
fn extract_log_entry(doc: &TantivyDocument, schema: &LogSchema) -> Option<LogEntry> {
    let mut missing_fields = Vec::new();
    let mut result = LogEntry { ... };
    // 收集所有缺失字段，统一处理
}
```

#### 优化3: SnippetCacheKey稳定性
```rust
// 当前: Debug格式不稳定
doc_id: format!("{:?}", doc_address),

// 优化: 使用确定性格式
doc_id: format!("seg{}_off{}", doc_address.segment_ord, doc_address.doc_id),
```

### 5.2 中期优化（1-2周）

#### 优化4: 预计算时间分区Bitmap
- **问题**: query_time_range 遍历所有分区
- **方案**: 预计算小时级分区Bitmap，查询时直接合并
- **预期收益**: 查询时间减少50%+

#### 优化5: 并行文档转换
- **问题**: document_to_log_entry 串行执行
- **方案**: 使用rayon并行处理文档数组
- **预期收益**: 批量查询速度提升3-5倍

#### 优化6: TermStats LRU缓存
- **问题**: HashMap无淘汰机制
- **方案**: 改用LruCache并定期持久化
- **预期收益**: 内存使用减少60%+

### 5.3 长期优化（>1月）

#### 优化7: 查询计划缓存
- **问题**: 相同查询重复解析
- **方案**: 实现QueryPlan Cache with TTL
- **预期收益**: 查询延迟减少20%

#### 优化8: 向量化搜索
- **问题**: 当前标量搜索
- **方案**: 集成向量搜索引擎（如Meilisearch）
- **预期收益**: 语义搜索能力
