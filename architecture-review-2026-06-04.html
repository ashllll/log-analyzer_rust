首要推荐：拆分 AppState God Object

AppState 是当前最大的架构摩擦源。它混合了 9 个不相关领域的运行时状态，每次新增功能都迫使修改者理解全部字段的交互关系。 将其按领域拆分为独立的上下文（WorkspaceContext / SearchContext / TaskContext），能立即降低认知负担， 并让 WorkspaceService 迁移（候选 #2）有清晰的落地位置。

AppState — God Object

STRONG
models/state.rs main.rs commands/*.rs application/*.rs
接口复杂度
9 个领域字段 + 15+ 方法
认知负担
修改任一功能需理解全部
测试难度
必须构造完整 AppState

问题

AppState 同时持有：workspace_services、watchers、search_cancellation_tokens、search_metrics、 disk_result_store、search_thread_pool、task_manager、state_sync、async_resource_manager。 这些属于 5 个完全不同的领域（Workspace、Search、Task、Sync、Resource），却被迫塞进一个结构体。 删除测试：删除 AppState 不会集中复杂度，而是迫使每个调用者自己组装所需依赖 → 这是一个浅层的胶水层，不是深模块。

Before
pub struct AppState {
    workspace_services: Arc>,
    disk_result_store: RwLock,
    task_manager: Arc,
}
After
pub struct AppState {
    workspaces: WorkspaceContext,
    search: SearchContext,
    tasks: TaskContext,
}

pub struct WorkspaceContext {
    services: HashMap,
    watchers: HashMap,
}

pub struct SearchContext {
    metrics: SearchMetrics,
    disk_store: DiskResultStoreRef,
    thread_pool: Arc,
}

收益

  • Locality：Workspace 相关修改只触及 WorkspaceContext，不污染 SearchContext
  • 测试：测试搜索逻辑只需构造 SearchContext（5 个字段），而非完整 AppState（15+ 字段）
  • Leverage：每个上下文的接口从"所有字段"缩小到"该领域需要的字段"

WorkspaceService 迁移 — 半途而废

STRONG
application/workspace_service.rs infrastructure/workspace_service_impl.rs commands/import.rs commands/watch.rs

问题

WorkspaceService trait 定义了 3 个服务接口（SearchService / ImportService / WatchService）， 但只有 SearchService 有实现。ImportService 和 WatchService 是 todo!() 占位。 更严重的是，WorkspaceService trait 暴露了 cas()、metadata_store()、search_engine() 三个内部依赖 getter（标注为"P5 渐进式，待清理"），这让调用者直接绕过服务接口访问底层存储。

结果：命令层和 WorkspaceServiceImpl 之间形成了"一半新一半旧"的混合状态。 导入命令仍然直接操作 CAS 和 MetadataStore，而不是通过 WorkspaceService。 这违背了"预组装服务"的设计初衷——调用者仍需了解底层依赖。

Before
// commands/import.rs
let cas = ContentAddressableStorage::new(...);
let metadata = MetadataStore::new(...);
// 直接操作底层存储，不经过 WorkspaceService

trait WorkspaceService {
    fn cas(&self) -> &Arc; // P5 待清理
    fn metadata_store(&self) -> &Arc; // P5 待清理
}
After
// WorkspaceService trait 移除内部 getter
trait WorkspaceService: SearchService + ImportService + WatchService {
    fn workspace_id(&self) -> &str;
}

// commands/import.rs
let ws = app.get_workspace_service(id)?;
ws.import_file(path, options).await?; // 通过接口

收益

  • Seam 完整性：WorkspaceService 成为真正的 seam，调用者只需知道接口
  • Locality：导入逻辑集中在 ImportService 实现中，不在命令层分散
  • 测试：可以 mock WorkspaceService 来测试命令层，无需构造真实 CAS/MetadataStore

SearchUseCase::execute_blocking — 闭包地狱

WORTH EXPLORING
application/search.rs infrastructure/searcher.rs

问题

execute_blocking 有 10+ 个参数，内部进行了 7 次 Arc::clone 和大量 move。 最严重的问题：在 spawn_blocking 的同步上下文中使用 tokio::runtime::Handle::current().block_on(events.emit_...()) 来发射事件——这是在阻塞线程池中进行异步阻塞调用，形成双重阻塞。

批处理 flush 逻辑（batch_size、results_count、was_truncated、timed_out）全部内联在一个函数中， 无法独立测试。取消检测、超时检测、max_results 截断逻辑交错在一起，阅读者需要同时跟踪 4 个状态变量。

Before
fn execute_blocking(
    log_files, results, events, searcher, // 4 Arc
    search_id, query, terms, filters, files,
    max_results, thread_pool, token,      // +3
) {
    // 7x Arc::clone
    // tokio::Handle::current().block_on(events.emit_...)
    // 内联批处理 + 取消 + 超时 + 截断逻辑
}
After
struct SearchOrchestrator { ... }

impl SearchOrchestrator {
    fn run(&self, ctx: SearchContext) -> SearchOutcome {
        let results = self.search_files(ctx.files)?;
        self.result_writer.write_in_batches(results)?;
        SearchOutcome { count, duration, truncated }
    }
}

// 事件发射通过 bounded channel，不 block_on

收益

  • 移除 block_on 反模式：使用 bounded channel 或事件队列在 spawn_blocking 和 async 上下文间传递事件
  • 可测试的批处理：ResultWriter 成为独立模块，可单独测试分页、超时、截断逻辑
  • Locality：取消/超时/截断策略集中在 Orchestrator 中，不再散落在闭包里

commands/search/mod.rs — 职责混合

WORTH EXPLORING
commands/search/mod.rs commands/search/query.rs commands/search/filters.rs

问题

这个 250+ 行的文件同时承担：参数验证（validate_search_params）、 运行时配置加载（load_search_runtime_config — 直接读取文件系统）、 workspace ID 解析（resolve_workspace_id）、取消令牌生命周期管理、 SearchUseCase 调用协调，以及 3 个 Serde 类型定义（BinarySearchResult、 BinarySearchRequest、SearchRuntimeConfig）。

load_search_runtime_config 直接读取 config.json，没有通过 AppConfigProvider trait — 违反了"Domain Trait 模式"的设计决策。 这意味着测试时无法注入 mock 配置，必须操作真实文件系统。

Before
// commands/search/mod.rs (250+ lines)
fn validate_search_params(...)      // 验证
fn load_search_runtime_config(...)  // 直接读文件
fn resolve_workspace_id(...)        // workspace 解析
fn cancel_search(...)               // 取消命令
fn fetch_search_page(...)           // 分页命令
fn search_logs(...)                 // 搜索命令
// + 3 个 struct 定义
After
// commands/search/mod.rs (仅命令入口)
#[command] pub async fn search_logs(...) -> Result<...>

// 配置加载移至 infrastructure/
fn load_search_runtime_config(
    provider: &dyn AppConfigProvider  // 通过 trait
) -> SearchRuntimeConfig

// 参数解析提取到独立模块
// commands/search/params.rs

收益

  • 接口一致性：配置加载通过 AppConfigProvider trait，支持 mock 注入
  • Locality：验证逻辑集中在 validation 模块，不再和命令入口混在一起
  • 测试：搜索命令测试可以用内存配置，不需要读写真实文件

QueryEngineLogSearcher — 隐式状态耦合

WORTH EXPLORING
infrastructure/searcher.rs services/query_planner.rs services/query_executor.rs

问题

build_plan 将 plan 存入 self.plans: Mutex>， match_content 再通过 u64 key 取出来。这个 HashMap只增不减——每次不同查询都会新增条目，没有清理机制，形成事实上的内存泄漏。

更严重的是，match_content 中重复编译 CompiledSearchFilters： SearchUseCase::execute 已经编译过一次 filters 并传给 get_files_with_filters， 但 QueryEngineLogSearcher::match_content 又重新编译了一次。 两层过滤（CompiledSearchFilters 在 SearchUseCase 和 QueryEngineLogSearcher 中各出现一次） 让调用者不清楚过滤到底发生在哪一层。

Before
impl LogSearcher for QueryEngineLogSearcher {
    fn build_plan(&self, query) -> DomainExecutionPlan {
        let plan = self.builder.build(query)?;
        let id = hash(query);
        self.plans.lock().insert(id, plan); // 只增不减！
        DomainExecutionPlan { id, ... }
    }

    fn match_content(&self, ..., plan: &DomainExecutionPlan, ...) {
        let compiled = CompiledSearchFilters::compile(filters)?; // 重复编译
        // 再次过滤
    }
}
After
impl LogSearcher for QueryEngineLogSearcher {
    fn build_plan(&self, query) -> (DomainExecutionPlan, ServicePlan) {
        let service_plan = self.builder.build(query)?;
        let domain_plan = DomainExecutionPlan::from(&service_plan);
        (domain_plan, service_plan) // 显式返回，不隐式存储
    }

    fn match_content(&self, ..., plan: &ServicePlan,
                    compiled_filters: &CompiledFilters) {
        // 使用调用者传入的已编译 filters
    }
}

收益

  • 消除内存泄漏：plans HashMap 不再需要，每次查询的 plan 随调用栈释放
  • 消除重复编译：CompiledSearchFilters 编译一次，传递给 match_content
  • 明确职责：过滤发生在 SearchUseCase 层，match_content 只做匹配

前端 Hooks — 过度拆分

SPECULATIVE
src/hooks/*.ts src/pages/SearchPage/hooks/*.ts src/services/api.ts

问题

全局有 20+ 个 hooks，SearchPage 单独就有 7 个（useSearchState, useSearchQuery, useSearchPagination, useSearchEvents, useVirtualScroll, useWorkspaceTimeRange）。 但很多 hooks 只被一个组件使用，没有复用价值。 useInfiniteSearch、useSearchListeners、useServerQueries 之间存在职责重叠—— 都在处理搜索相关的状态和事件，却分散在三个文件中。

api.ts 中的 LogAnalyzerApi 类封装了所有后端调用，但每个方法都是一个 thin wrapper。 这个 600+ 行的类本可以用一个泛型辅助函数大幅缩减。

Before
// SearchPage 使用 7 个 hooks
const state = useSearchState();
const query = useSearchQuery();
const pagination = useSearchPagination();
const events = useSearchEvents();
const scroll = useVirtualScroll();
const timeRange = useWorkspaceTimeRange();
// + useInfiniteSearch 在父级
After
// 合并为 SearchPage 专用 hook
const search = useSearchPage({ workspaceId, query });
// search.state, search.pagination, search.scroll, ...

// 或按"关注点"而非"功能"拆分：
// useSearchData (数据获取)
// useSearchUI (UI 状态)

收益

  • 减少文件跳转：理解搜索页状态管理不需要打开 7 个文件
  • 减少隐式耦合： hooks 间的隐式依赖（通过 Zustand store）更可控
  • Leverage：按"页面级"或"关注点"拆分，比按"功能原子"拆分更能反映业务意图
