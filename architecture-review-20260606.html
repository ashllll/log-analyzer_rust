1. 完成 WorkspaceService 迁移

涉及文件

  • application/workspace_service.rs — trait 定义
  • infrastructure/workspace_service_impl.rs — 实现
  • models/state.rs — AppState 容器
  • commands/search/mod.rs — 搜索命令
  • commands/import.rs — 导入命令

问题

代码库正处于从「运行时 HashMap 服务定位器」向「预组装 WorkspaceService」模式的迁移中途。 workspace_service.rs:17-23 的迁移状态注释标明 P1（SearchService 实现）、P2（AppState 集成）、P3（search_logs 迁移）、P6（清理旧 HashMap）均未完成。 当前 两套架构并存：部分路径走 WorkspaceService，部分仍直接访问 AppState 的原始 HashMap。

方案

完成迁移。AppState 仅持有 WorkspaceServiceRef 实例——不再有原始的 cancellation_tokens HashMap， 不再暴露存储引用。所有命令统一通过 WorkspaceService 操作，每个 WorkspaceServiceImpl 拥有其工作区的完整运行时状态（CAS、MetadataStore、SearchEngine、DiskResultStore、搜索会话、监听器状态）。

收益

杠杆

调用方通过一个 WorkspaceService 接口获取工作区全部能力，而非管理 5+ 个独立 Arc 引用

局部性

一个工作区的运行时状态全部集中在 WorkspaceServiceImpl 内

可测试性

Mock WorkspaceService trait 即可测试命令层，无需构造持有 6 个字段的 AppState

AI 可导航性

单一架构模式，AI 无需在两条代码路径之间切换推理

Worth exploring

2. 合并查询流水线层

涉及文件

  • services/query_validator.rs — ~100行
  • services/query_planner.rs — ~400行
  • services/query_executor.rs — ~130行
  • infrastructure/searcher.rs — 实际执行

问题

查询流水线有三个浅模块串行排列： QueryValidator（一个静态方法 validate()）、 QueryPlanner（构建 ExecutionPlan）、 QueryPlanBuilder（在 Planner 外加一层 plan_cache，再委托给 Validator）。 删除测试：删除 QueryPlanBuilder，其缓存逻辑移到 Planner 内部——无复杂性扩散。 删除 QueryValidator，其验证逻辑移到 Planner 的 build_plan 开头——验证是规划的前置条件，本来就在同一调用链上。

方案

将 QueryValidator 内联到 QueryPlanner（验证是规划的前置条件），将 QueryPlanBuilder 合并到 QueryPlanner（缓存是实现细节）。 结果：一个深模块 QueryPlanner，接口是 build_plan(query) → ExecutionPlan， 内部处理验证、引擎选择、缓存——调用方无需了解内部步骤。

调用流程对比

Before — 三层浅调用

sequenceDiagram participant CMD as Command participant PB as QueryPlanBuilder participant QV as QueryValidator participant QP as QueryPlanner CMD->>PB: build(query) PB->>QV: validate(query) QV-->>PB: result PB->>PB: cache check PB->>QP: build_plan(query) QP-->>PB: ExecutionPlan PB-->>CMD: ExecutionPlan

After — 一个深模块

sequenceDiagram participant CMD as Command participant QP as QueryPlanner CMD->>QP: build_plan(query) QP->>QP: validate (inline) QP->>QP: cache lookup QP->>QP: engine select + compile QP-->>CMD: ExecutionPlan

收益

深度

接口从 3 个模块 × 1-2 个公开方法变为 1 个模块 × 1 个方法，接口复杂度降低 ~70%

局部性

修改查询构建逻辑时只需打开一个文件。验证规则的变更不再需要跨文件追踪

接口即测试面

一个 build_plan 方法覆盖验证+规划+缓存，测试更集中

Worth exploring

3. 深化 WorkspaceRepo

涉及文件

  • infrastructure/workspace_repo.rs — ~50行
  • infrastructure/workspace_service_impl.rs — 调用方
  • application/workspace_service.rs — re-exports

问题

删除测试：删除 WorkspaceRepo，其唯一调用方 WorkspaceServiceImpl 改为直接持有 4 个 Arc 引用。 WorkspaceRepo 不添加任何行为——只有 4 个 getter 方法。没有复杂性集中——删除后复杂性不会扩散到 N 个调用方。 这是一个纯透传模块，接口和实现复杂度几乎相同。

方案

给 WorkspaceRepo 注入行为：吸收工作区路径解析、元数据完整性校验、搜索结果会话管理。 接口不再是 4 个 getter，而是 get_file_by_hash()、verify_integrity()、 manage_search_sessions() 等有杠杆的方法。

收益

杠杆

调用方通过 WorkspaceRepo 完成工作区数据操作，而非自行组合 4 个底层存储引用

接缝

如果工作区数据存储方式变更（SQLite → Tantivy），只需修改 WorkspaceRepo

Speculative

4. 统一领域 trait 位置

涉及文件

  • crates/la-core/src/domain/ — 7 个 trait
  • crates/la-core/src/traits.rs — 5 个 trait
  • 所有 use la_core::domain 和 use la_core::traits

问题

领域 trait 分散在两个位置：la-core/src/domain/（7 个 trait） 和 la-core/src/traits.rs（5 个 trait）。 这个分裂是任意的——ContentStorage 和 MetadataStorage 明显是领域抽象（CAS 存储、元数据索引），在 CONTEXT.md 中明确定义。 新贡献者（包括 AI）需要发现这个分裂——他们找到了 domain/ 后可能以为那就是全部。

方案

将所有领域 trait 移入 la-core/src/domain/，按主题分子模块： storage.rs（ContentStorage, MetadataStorage）、 search.rs（LogSearcher, LogFileRepository, SearchResultRepository）、 event.rs、extract.rs、workspace.rs、task.rs。 AppConfigProvider 移到 config.rs。删除 traits.rs。

5. 减少 LogSearcher 适配器间接层

涉及文件

  • crates/la-core/src/domain/ — LogSearcher trait
  • infrastructure/searcher.rs — 唯一实现
  • application/search.rs — SearchUseCase

问题

LogSearcher trait 只有一个适配器：QueryEngineLogSearcher。 一个适配器 = 假设的接缝。4 个方法签名精确镜像唯一实现。接口和实现的复杂度几乎相同——浅 trait。 增加了动态分发（Arc<dyn LogSearcher>）但未提供接缝的好处。

方案

路径 A：找到第二个适配器（如 TantivyOnlySearcher），使接缝成为真实接缝。
路径 B：删除 trait，直接使用 QueryEngineLogSearcher 具体类型。

简单性

减少一层动态分发，依赖图更扁平，代码导航更直接

或：真实接缝

如果走路径 A，第二个适配器带来真正的可替换性

首选推荐

#1 — 完成 WorkspaceService 迁移

这是最紧迫的候选。两套架构并存直接影响每个新功能的实现路径选择。 workspace_service.rs 的迁移注释标明 4 个阶段未完成，技术债务正在积累。

  • AppState 简化为仅持有 WorkspaceServiceRef 映射
  • 工作区运行时状态全部由 WorkspaceServiceImpl 拥有——局部性大幅提升
  • 命令层测试只需 mock WorkspaceService trait
  • 为候选 #2 和 #3 铺平道路

预计影响：~5 文件、~200 行变更。风险：低——trait 已定义，实现已部分完成。本质上是完成已经开始的重构。
